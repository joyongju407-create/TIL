[26.02.05] GitHub 협업 Flow, 재귀함수, 그리고 충돌 해결
1. 💡 Today I Learned (배운 내용)
A. Git & GitHub 협업 프로세스 (GitHub Flow)

기본 흐름: 이슈 생성(Issue) → 브랜치 생성(Branch) → 작업 및 커밋 → PR(Pull Request) → Merge → 브랜치 삭제.

이슈 연결: PR 작성 시 resolve #이슈번호를 적으면 Merge 시 해당 이슈가 자동으로 닫힌다.

동기화: 원격 저장소(GitHub)에서 Merge 된 후에는 반드시 로컬에서도 git pull origin main으로 최신 상태를 맞춰야 한다.

B. Git 충돌(Conflict) 해결의 정석

공통 과정: 충돌 발생 시 에디터에서 <<<<, ====, >>>> 기호를 지우고 원하는 코드만 남긴 후 git add .을 한다.

Merge 상황: 수정 후 **git commit**을 입력하면 합쳐지며 완료된다.

Rebase 상황: 수정 후 **git rebase --continue**를 입력해야 다음 단계로 넘어간다. (망했을 땐 git rebase --abort로 탈출)

C. 파이썬 재귀함수 (Recursion)와 최적화

재귀의 의미: fibo(n-1) + fibo(n-2)는 단순 덧셈이 아니라, 작업을 쪼개서 바닥(Base Case)까지 내려갔다 합쳐서 올라오는 분할 정복 구조다.

메모이제이션(Memoization): 재귀의 중복 계산 문제를 해결하기 위해 딕셔너리(pad)에 정답을 적어두고(Caching) 재사용하는 기법.

D. 개발 효율 도구

pre-commit: 커밋 전 자동으로 코드를 검사해주는 도구. 설정 파일(.pre-commit-config.yaml)은 프로젝트 최상위 루트에 있어야 한다.

커밋 컨벤션: Feat(기능), Fix(버그), Docs(문서) 등 말머리를 달아 작업 내용을 명확히 한다.

2. 💦 Trouble & Shooting (오늘의 삽질과 해결)
Q1. Git 명령어가 기억나지 않아 흐름이 끊겼다.

해결: 무작정 외우기보다 **"생성(switch -c) → 작업 → 저장(commit) → 합치기(merge/push)"**라는 행동의 순서를 먼저 떠올리며 명령어를 매칭시켰다.

Q2. 마크다운 체크박스가 렌더링 되지 않았다.

원인: -[] 처럼 괄호 안에 공백을 넣지 않아서 발생함.

해결: - [ ] 처럼 대괄호 안에 반드시 띄어쓰기를 해야 함을 알았다. (체크된 건 - [x])

Q3. 마크다운 링크 문법 순서가 헷갈렸다.

원인: (이름)[주소]인지 [이름](주소)인지 혼동됨.

해결: "대괄호에는 이름, 소괄호에는 주소" 공식으로 암기했다. ([Google](https://google.com))

3. 📝 Code & Command Cheatsheet
Git 필수 명령어

Bash
# 브랜치 생성 및 이동
$ git switch -c feature/새기능

# 작업 후 원격 업로드 (최초 1회)
$ git push -u origin feature/새기능

# 병합 후 로컬 동기화 (필수)
$ git switch main
$ git fetch origin main
$ git merge FETCH_HEAD (또는 git pull origin main)
$ git branch -d feature/새기능
피보나치 메모이제이션 구현 (Python)

Python
pad = {0: 0, 1: 1} # 정답 노트(Cache)
def fibo(n):
    if n not in pad:
        # 없으면 계산해서 적어놓기
        pad[n] = fibo(n - 1) + fibo(n - 2)
    return pad[n] # 적혀있는 답 제출
